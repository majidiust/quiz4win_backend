# WingoBingo / Quiz4Win Real-Time Quiz Backend Architecture

## 1. Purpose

This document defines the backend architecture for a real-time multilingual quiz game where questions are generated by an LLM, broadcast to mobile clients, answered within a strict time window, and validated with high consistency, integrity, security, and availability.

The target clients are Android and iOS applications. Some games may have a live stream, and some may run without video streaming.

The architecture is designed for:

- Real-time question delivery
- Strict 10-second answer windows
- Multilingual question display
- Server-side answer validation
- Anti-cheat protection
- Fast user elimination
- High availability
- Auditability
- Replay and duplicate protection
- Support for automatic and manual/live-show game modes

---

## 2. Core Architectural Decision

### 2.1 Main Rule

Redis is the real-time source of truth during an active game.

The database is the persistent source of truth before and after the game.

LiveKit/WebSocket is used for broadcasting real-time events.

API or Secure WebSocket Gateway is used for answer submission.

RabbitMQ/Kafka is used only for internal service communication.

FCM is used only for notifications and should not be used for real-time gameplay.

---

## 3. High-Level Components

### 3.1 Game Scheduler

Responsible for detecting games that should start.

Responsibilities:

- Runs every minute or at a configured interval
- Checks scheduled games
- Finds games whose start time has arrived
- Emits `StartGame` command to RabbitMQ
- Does not generate questions
- Does not broadcast questions
- Does not validate answers

### 3.2 Game Orchestrator

The main brain of the game session.

Responsibilities:

- Starts game sessions
- Prepares Redis state
- Controls game lifecycle
- Controls question lifecycle
- Handles auto mode and manual/event mode
- Coordinates question generation
- Broadcasts game events
- Closes questions
- Finalizes results
- Recovers running games after restart

### 3.3 Question Engine

Responsible for generating quiz questions using OpenAI or another LLM provider.

Responsibilities:

- Receives topic, category, difficulty, question type, target languages, and game constraints
- Generates one canonical question
- Generates localized versions for requested languages
- Ensures stable option IDs across all languages
- Produces exactly one correct answer
- Returns metadata and validation flags
- Does not directly decide player results

### 3.4 Translation / Localization Engine

Can be part of the Question Engine or a separate service.

Responsibilities:

- Localizes question text and options
- Keeps option IDs stable
- Ensures correct option meaning remains identical across languages
- Supports dynamic target language list

### 3.5 Broadcast Gateway

Responsible for sending real-time events to clients.

Recommended options:

- LiveKit DataChannel when the game has a LiveKit room
- Dedicated WebSocket when the game does not have video/live stream
- Language-specific rooms/topics/channels if needed

Used for:

- Game started
- Question started
- Timer sync
- Question closed
- User eliminated
- Leaderboard updates
- Game ended
- Answer reveal after question close

Not used as the primary secure answer submission channel.

### 3.6 Answer Gateway

Responsible for receiving answer submissions from clients.

Recommended options:

- Fast HTTPS API
- Secure WebSocket Gateway for ultra-low latency
- API fallback if WebSocket fails

Responsibilities:

- Receives user answer
- Authenticates user/session
- Checks game and user state in Redis
- Performs atomic validation
- Returns immediate result
- Emits answer event for async persistence

### 3.7 Redis

Real-time source of truth for active games.

Stores:

- Game status
- Current question
- Question start/end time
- Correct option ID
- User participant/spectator/eliminated status
- User answer history
- Wrong answer count
- Remaining lives
- Duplicate answer locks
- Attempt IDs
- Active sessions
- Temporary leaderboard
- Join status
- Eligibility rules

### 3.8 Database

Persistent source of truth.

Stores:

- Game definitions
- Game schedule
- Game rules
- Redis namespace/cluster reference
- Question records
- User participation records
- Answer logs
- Final results
- Audit logs
- Prize distribution records

Important: Do not store raw Redis passwords or sensitive connection secrets in the database.

Store only:

- `redisClusterId`
- `redisNamespace`
- `stateStatus`
- `startedAt`
- `expiresAt`

### 3.9 RabbitMQ / Kafka

Internal command/event bus.

Used for:

- StartGame command
- GenerateQuestion command
- BroadcastQuestion command
- CloseQuestion command
- PersistAnswer event
- FinalizeGame event
- Operator/AI host commands

Not used directly by mobile clients.

## 4. Game Modes

### 4.1 Auto Mode

Used when the game runs automatically.

Flow:

1. Game starts
2. Orchestrator generates/prepares question
3. Question is stored in Redis
4. Question is broadcast
5. Answer window opens
6. After configured time, question closes
7. Next question starts automatically
8. Game ends when question pool is finished or game rules trigger ending

### 4.2 Manual / Event Mode

Used for live shows, operator-controlled shows, or AI-hosted shows.

Flow:

1. Game starts
2. Orchestrator waits for command
3. Operator / AI Host / Admin Panel sends command through RabbitMQ
4. Orchestrator prepares question
5. Question is stored in Redis
6. Question is broadcast
7. Question closes after time limit
8. Orchestrator waits for next command

## 5. LLM Question Generation

### 5.1 Dynamic Inputs

The Question Engine should accept:

- `topic`
- `category`
- `difficulty`
- `questionType`
- `targetLanguages`
- `numberOfOptions`
- `audienceLevel`
- `regionOrCulture`
- `avoidTopics`
- `timeLimitSeconds`
- `gameType`

### 5.2 Suggested Question Types

Supported question types:

- Factual
- Reasoning
- Comparison
- Scenario-based
- Definition
- Sequence
- True/False
- Speed question
- Elimination question

Recommended for first version:

- Factual
- Simple reasoning
- Comparison
- Short scenario-based

### 5.3 Difficulty Levels

Easy:

- General knowledge
- Fast answer
- No deep reasoning

Medium:

- Requires some knowledge or simple comparison

Hard:

- Requires more specific knowledge or quick reasoning

### 5.4 LLM Output Requirements

The LLM must produce:

- One canonical question identity
- Four stable options
- One correct option ID
- Localized question text for each requested language
- Localized option text for each requested language
- Short explanation
- Estimated answer time
- Safety validation flags
- Ambiguity validation flags

### 5.5 Multilingual Rule

The question identity, correct option, and option IDs must remain identical across all languages.

Only displayed text changes.

Example:

- Option A must represent the same concept in English, Arabic, Persian, Turkish, German, Italian, etc.
- Correct answer must be the same option ID in every language.
- The client must submit only `questionId` and `selectedOptionId`, not answer text.

### 5.6 LLM Safety Requirements

Questions must avoid:

- Political manipulation
- Hate speech
- Sexual content
- Religious provocation
- Illegal instructions
- Ambiguous or misleading wording
- Multiple correct answers
- Regionally offensive phrasing
- Unverifiable claims
- Questions impossible to answer in the configured time limit

### 5.7 Pre-Generation Requirement

Questions should be generated before they are needed.

For a 10-second game flow, the system should never depend on live LLM latency during the active question window.

Recommended approach:

- Generate at least several questions ahead
- Validate and localize them before broadcast
- Store active and upcoming questions in Redis
- Persist final question records in DB asynchronously or before game starts

- `activeConnectionStatus`
- `eliminatedAt`
- `eliminationReason`

## 7. Join Rules

### 7.1 Main Join Policy

A user can participate only if they join during the first question while the first question is still active.

If the first question is already closed, the user becomes spectator only.

### 7.2 Entering During First Question

If the user enters during the first question:

- The user can participate
- Remaining time is calculated by server
- The user sees only remaining time
- Example: if the question lasts 10 seconds and user joins after 3 seconds, they have 7 seconds left

### 7.3 Entering After First Question

If even one question has already passed:

- The user becomes spectator
- The user can see questions and options
- The user cannot submit answers
- Any answer submission must be rejected by backend

### 7.4 Join API Responsibilities

When user joins, backend must return:

- `userStatus`
- `gameStatus`
- `currentQuestionIndex`
- `currentQuestionId`
- `serverTime`
- `questionEndsAt`
- `remainingTimeMs`
- `canSubmitAnswer`
- localized payload based on user language

### 7.5 Server-Side Enforcement

The client UI is not trusted.

Even if the client displays submit button incorrectly, the backend must enforce:

- User must be participant
- User must have joined before first question closed
- User must not be eliminated
- Current question must be active
- Answer must arrive before `endsAt + gracePeriod`

## 8. Question Broadcast Flow

### 8.1 Correct Flow

1. Question Engine generates question
2. Translation/localization is completed
3. Orchestrator validates question
4. Orchestrator writes question state to Redis
5. Orchestrator sets `startsAt` and `endsAt`
6. Orchestrator broadcasts `QUESTION_STARTED`
7. Clients display localized question
8. Clients submit answers through Answer Gateway
9. Orchestrator closes question after time limit
10. Correct answer may be revealed only after question closes

### 8.2 Important Rule

Never broadcast a question before it is registered in Redis.

Redis must be ready before clients can answer.

---

## 9. Answer Submission Flow

### 9.1 Client Submission

Client sends:

- `gameId`
- `questionId`
- `selectedOptionId`
- `attemptId`
- `clientSentAt` only for analytics, not for validation

Client must not send:

- Answer text
- Correct answer
- Local decision
- Client timer result as truth

### 9.2 Backend Validation

Answer Gateway checks:

- Authentication
- User session
- Device/session validity
- Game exists
- Game is active
- Redis context exists
- User is participant
- User is not spectator
- User is not eliminated
- Question is active
- Current question matches submitted question
- Server receive time is inside valid answer window
- Attempt is not duplicate
- User has not already answered this question
- Selected option is valid
- Selected option equals correct option

### 9.3 Atomic Redis Validation

Validation must be atomic.

The system must not perform duplicate check, wrong count update, and elimination decision in separate unsafe operations.

Atomic validation must include:

- Duplicate answer lock
- Question active check
- User status check
- Time window check
- Correctness check
- Wrong count update
- Remaining lives update
- Elimination decision
- Answer result storage

This prevents:

- Race conditions
- Double answer
- Replay attack
- Multiple device cheating
- Duplicate packet problems

### 9.4 Immediate Response

Backend immediately returns:

- Accepted or rejected
- Correct or wrong
- User status after answer
- Remaining lives
- Server time
- Elimination status if applicable

### 9.5 Async Persistence

After immediate Redis validation:

- Emit answer event to RabbitMQ/Kafka
- Persist answer log in DB asynchronously
- Persist final score/result later

## 10. Wrong Answer and Elimination Rules

### 10.1 Game Rule

Each game defines:

- `maxWrongAnswers`

Example:

- `maxWrongAnswers = 1`

This means the user can make only one mistake depending on the exact product rule.

Recommended strict rule:

- If `wrongCount >= maxWrongAnswers`, eliminate the user
- Or if using lives model: if `remainingLives <= 0`, eliminate the user

The exact rule must be consistent across product, UI, and backend.

### 10.2 No Answer Rule

If the user does not answer within the active question window:

- It is counted as wrong
- The user may be eliminated if wrong limit is reached

### 10.3 Late Answer Rule

If the answer arrives after the allowed server-side window:

- It is counted as wrong
- The user may be eliminated
- Client timestamp is ignored

### 10.4 Disconnect Rule

If user disconnects and misses the answer window:

- It is treated as no answer
- No answer is counted as wrong
- If wrong limit is reached, user is eliminated

### 10.5 Reconnect Rule

If the user reconnects:

- If still participant, not eliminated, and current question is still active, they may continue with remaining server-side time
- If the answer window has passed, the missed question is counted as wrong
- If wrong limit is reached, the user is eliminated
- If the user joined after question 1 closed, they remain spectator only

### 10.6 Immediate Elimination

When a user is eliminated:

- Redis updates `userStatus = eliminated`
- Redis stores `eliminatedAt`
- Redis stores `eliminationReason`
- Backend response tells user they are eliminated
- Broadcast may notify relevant clients/system
- DB persistence happens asynchronously

## 11. Time Synchronization

### 11.1 Server Time Is the Only Truth

Mobile device time is not trusted.

Backend must provide:

- `serverTime`
- `questionStartsAt`
- `questionEndsAt`
- `remainingTimeMs`

Client only displays countdown.

### 11.2 Grace Period

A very small grace period may be configured.

Recommended:

- 300ms to 500ms maximum

Purpose:

- Network jitter tolerance

Not allowed:

- Long grace periods
- Client-side extension
- User-specific unfair extension

### 11.3 Late Packets

If a packet arrives late at backend:

- Backend receive time is used
- Client sent time is not used for validation
- Late answer is wrong or rejected based on game rule

---

## 12. Spectator Mode

### 12.1 Who Becomes Spectator

A user becomes spectator if:

- They join after question 1 closed
- They are not eligible
- They were eliminated but remain watching
- They enter without valid participation ticket if applicable

### 12.2 Spectator Permissions

Spectator can:

- Watch live game
- See questions
- See options
- See public leaderboard if enabled
- See final result

Spectator cannot:

- Submit answer
- Affect leaderboard
- Claim prize
- Rejoin as participant in the same game

---

## 13. Security Requirements

### 13.1 Client Is Not Trusted

Never trust:

- Client timer
- Client selected language as authority
- Client answer text
- Client-reported correctness
- Client join time
- Client sent timestamp
- Client local state

### 13.2 Required Protections

Implement:

- JWT/session authentication
- Device/session validation
- Rate limiting
- Attempt ID idempotency
- Redis atomic locks
- Single active session policy
- Replay protection
- Duplicate answer prevention
- Server-side time validation
- Private Redis network
- Service-to-service credentials
- Secure internal RabbitMQ credentials
- Structured audit logging

### 13.3 Single Active Session

Recommended rule:

- One user can have only one active participant session per game
- If user logs in from another device, either reject second session or invalidate previous session
- This decision must be explicit in product rules

### 13.4 Correct Answer Protection

Correct answer must never be sent to clients before question is closed.

Before close:

- Client receives only question text and options

After close:

- Correct answer may be broadcast for display

---

## 14. Consistency and Integrity

### 14.1 Consistency Model

During active game:

- Redis is the source of truth

After game ends:

- Database becomes source of truth

### 14.2 Integrity Rules

- Every answer must be linked to gameId, questionId, userId, attemptId
- Every answer is processed once
- Every user has one status at a time
- Every question has one active time window
- Every question has exactly one correct option
- Every localized option maps to the same canonical option ID
- Every elimination must have reason and timestamp
- Every final result must be derived from Redis state and verified before DB finalization

### 14.3 Idempotency

Each answer submission must include `attemptId`.

If same attempt arrives twice:

- Return same result
- Do not count twice
- Do not increment wrong count twice

### 14.4 Duplicate Answer

If user submits multiple different answers for same question:

- First valid server-received answer counts
- Later answers are rejected or ignored
- Audit log should record suspicious attempts if needed

---

## 15. High Availability

### 15.1 Redis

Recommended:

- Redis Cluster or highly available Redis setup
- Private network only
- Persistence settings appropriate for short-lived game state
- Monitoring for memory, latency, evictions
- Key TTLs to avoid stale state

### 15.2 Orchestrator Recovery

If Game Orchestrator restarts:

- It must read running games from DB
- It must check Redis state
- It must recover current game phase
- It must continue or safely finalize the game
- It must avoid duplicate broadcasts

### 15.3 RabbitMQ / Kafka

Recommended:

- Durable queues where needed
- Dead-letter queues
- Retry policies
- Idempotent consumers
- Correlation IDs
- Command/event separation

### 15.4 API / Answer Gateway

Recommended:

- Horizontally scalable
- Stateless except Redis access
- Load balanced
- Uses Redis for shared real-time state
- Rate limited
- Observability enabled

### 15.5 Broadcast Gateway

If LiveKit is unavailable:

- For stream games: show degraded state or reconnect
- For non-stream games: fallback to WebSocket if available
- Gameplay decisions must still be based on Redis, not broadcast success

## 18. Final End-to-End Flow

### 18.1 Game Start

1. Scheduler finds scheduled game
2. Scheduler sends `StartGame`
3. Orchestrator creates Redis namespace
4. Orchestrator stores Redis context reference in DB
5. Orchestrator initializes game state in Redis
6. Game status becomes running

### 18.2 Question Start

1. Question Engine generates or selects question
2. Localization is prepared for requested languages
3. Question is validated
4. Question state is written to Redis
5. `startsAt` and `endsAt` are set
6. Question is broadcast to clients
7. Clients display localized version

### 18.3 User Join

1. User opens game
2. Backend checks Redis game state
3. If question 1 is active, user becomes participant
4. If question 1 is closed or later question is active, user becomes spectator
5. Backend returns server time and remaining time

### 18.4 Answer Submit

1. User submits answer
2. Answer Gateway receives request
3. Redis atomic validation runs
4. Result is returned immediately
5. If wrong/no answer/late answer exceeds allowed limit, user is eliminated
6. Answer event is persisted asynchronously

### 18.5 Question Close

1. Orchestrator closes question based on server time
2. No further answers are accepted
3. Missing answers are counted as wrong
4. Eliminations are applied
5. Correct answer may be revealed
6. Next question starts or game ends

### 18.6 Game End

1. Orchestrator finalizes Redis state
2. Final results are calculated
3. Results are persisted to DB
4. Redis keys receive TTL or are archived
5. Prize logic runs if applicable

---

## 19. Non-Negotiable Rules

The following rules must always be enforced:

1. Server time is the only source of truth.
2. Redis is the real-time source of truth during active games.
3. Database is the persistent source of truth after game finalization.
4. Question must be stored in Redis before broadcast.
5. Correct answer must not be sent before question close.
6. User answer validation must be atomic in Redis.
7. Client timer is not trusted.
8. Client answer text is not trusted.
9. Late answer is wrong.
10. No answer is wrong.
11. Disconnect during answer window may cause wrong/no-answer and elimination.
12. Join after question 1 is closed means spectator only.
13. Spectator cannot submit answers.
14. Duplicate answer must not be counted twice.
15. One user should not have multiple active participant sessions in the same game.
16. LLM questions must be validated before use.
17. Option IDs must remain stable across all languages.
18. Final results must be derived from Redis and persisted to DB.
19. Sensitive Redis credentials must not be stored in DB.
20. All critical events must be auditable.

## 21. Executive Summary

The recommended backend architecture separates real-time display from secure answer validation.

Questions are generated by an LLM, localized into dynamic languages, validated, written into Redis, and then broadcast to users. Users submit answers through a secure backend path. The backend validates answers atomically against Redis using server time. Wrong, late, missing, duplicate, or invalid answers are handled immediately. Users are eliminated in real time based on game rules.

LiveKit/WebSocket is used for real-time broadcasting. API or Secure WebSocket is used for answer submission. RabbitMQ/Kafka is used for internal service communication. Redis is the source of truth during the game. The database is used for persistence, audit, and final results.

This architecture provides consistency, integrity, security, low latency, and high availability for real-time quiz gameplay.
